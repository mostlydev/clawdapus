# Nullclaw Driver Implementation Plan

**Date:** 2026-02-28 (reviewed 2026-03-01)  
**Status:** Draft (revised for implementation readiness)  
**Goal:** Add a Clawdapus driver for `nullclaw` that is small, fail-closed, and extensible via `CONFIGURE nullclaw config set ...` directives parsed by the driver.

**Reference:** https://github.com/nullclaw/nullclaw

## Summary

Implement a new `internal/driver/nullclaw` package with the standard driver interface:

- `Validate`: reject invalid runtime contracts before startup.
- `Materialize`: write `config.json` + mounts + health config.
- `PostApply`: verify container running and register `INVOKE` jobs via `nullclaw cron add`.
- `HealthProbe`: report service health from driver perspective.

Primary design rule: map the common Clawdapus surfaces natively, and leave full schema flexibility to `CONFIGURE` directives (driver-parsed JSON-path setters).

## Scope

- In scope:
  - Nullclaw driver package (`config.go`, `driver.go`, tests)
  - Driver registration in build-time and health commands
  - INVOKE execution using nullclaw CLI in `PostApply`
- Out of scope:
  - Full channel surface policy parity beyond core HANDLE mapping
  - Advanced nullclaw health diagnostics
  - Migration tooling from other claw types

## Key design decisions

### 1) Keep config generation minimal and deterministic

- Generate only essential keys.
- Preserve deterministic JSON output (`json.MarshalIndent` + sorted provider sets).
- Apply precedence as: generated defaults < HANDLE defaults < CONFIGURE overrides.

### 2) Use nullclaw CLI for scheduled jobs

- `INVOKE` jobs are created in `PostApply` using `nullclaw cron add`.
- Avoid writing internal cron state files directly.
- Treat cron registration as a core behavior: job creation failures should fail `claw up`.

Important upstream constraint: `nullclaw cron add` accepts positional args (`<expression> <command>`) and stores shell commands, not structured agent payloads. The driver must translate `Invocation.Message` into a shell command (for example `nullclaw agent -m ...`).

### 3) Validate only what is required to start safely

- Require `AgentHostPath` and file existence.
- For known HANDLE platforms, require platform token env vars:
  - `discord` -> `DISCORD_BOT_TOKEN`
  - `telegram` -> `TELEGRAM_BOT_TOKEN`
  - `slack` -> `SLACK_BOT_TOKEN`
- Unknown HANDLE platforms: warn and continue (configurable later through `CONFIGURE`).

### 4) Avoid brittle health checks

- Upstream exposes `GET /health` as a public endpoint in gateway mode.
- Upstream runtime image installs `curl`; use a curl-based healthcheck where possible.
- `HealthProbe` should still include a container-running fallback to avoid false negatives from command/tooling drift.

### 5) Keep CONFIGURE semantics compatible with existing Clawdapus patterns

- Nullclaw does not currently expose a top-level `nullclaw config set` CLI command.
- Keep `CONFIGURE nullclaw config set <path> <value>` as a Clawdapus-side DSL, parsed and applied directly into generated `config.json`.

## Config mapping (initial)

| Clawdapus concept | Nullclaw path |
|---|---|
| `MODEL primary` | `agents.defaults.model.primary` |
| `MODEL fallback` | `reliability.fallback_providers` |
| `HANDLE discord` | `channels.discord.accounts.main.{bot_token,guild_id}` |
| `HANDLE telegram` | `channels.telegram.accounts.main.{bot_token,allow_from}` |
| `HANDLE slack` | `channels.slack.accounts.main.{bot_token,app_token?,signing_secret?,mode?}` |
| `CLLAMA` rewrite | `models.providers.<provider>.{base_url,api_key}` |
| gateway defaults | `gateway.{port:3000,host:"127.0.0.1",require_pairing:true}` |
| `CONFIGURE` | `nullclaw config set <path> <value>` (driver DSL), applied last |

Note: any path not covered above stays operator-configurable through `CONFIGURE`.

## Filesystem + runtime layout

```text
/root/.nullclaw/            # bind mount (rw): config + state
  config.json               # generated by driver
/claw/AGENTS.md             # agent contract (ro)
/claw/CLAWDAPUS.md          # generated infra context (ro)
/claw/skills/*              # mounted skills (ro)
/tmp                        # tmpfs
```

Runtime defaults in `MaterializeResult`:

- `ReadOnly: true`
- `Restart: "on-failure"`
- `SkillDir: "/claw/skills"`
- `SkillLayout: ""` (flat)
- `Environment: {"CLAW_MANAGED":"true"}`

## Implementation plan

### Task 1: Config generator (`config.go` + `config_test.go`)

Create `GenerateConfig(rc *driver.ResolvedClaw) ([]byte, error)` and test-first coverage for:

- gateway defaults are set
- model primary and fallback paths are correct
- cllama provider rewrites are applied for all providers referenced by models
- known HANDLE mappings write expected paths
- CONFIGURE command parsing + typed values (driver DSL, not nullclaw CLI)
- CONFIGURE overrides HANDLE defaults
- deterministic output across repeated calls

Required helper functions:

- `setPath(map[string]interface{}, string, interface{}) error`
- `parseConfigSetCommand(string) (path string, value interface{}, err error)`
- `resolveEnvTokenFromMap(map[string]string, string) string`
- `splitModelRef(string) (provider, model string, ok bool)`
- `collectProviders(map[string]string) []string` (sorted unique)

Acceptance gate:

- `go test ./internal/driver/nullclaw -v` passes.

### Task 2: Driver implementation (`driver.go` + `driver_test.go`)

#### Validate

- enforce agent contract presence/existence
- parse-validate all `CONFIGURE` commands
- enforce required env token per known HANDLE platform
- support optional Slack env passthroughs when present (`SLACK_APP_TOKEN`, `SLACK_SIGNING_SECRET`)
- warn for unknown HANDLE platforms
- accept `INVOKE` (handled in `PostApply`)

#### Materialize

- write generated config to `opts.RuntimeDir/nullclaw-home/config.json`
- mount `nullclaw-home` -> `/root/.nullclaw` (rw)
- mount contract + generated `CLAWDAPUS.md` to `/claw/...` (ro)
- return hardening defaults listed above
- include a healthcheck command valid for upstream image baseline (for example: `curl -fsS http://localhost:3000/health >/dev/null`)

#### PostApply

- fail if container ID missing or container not running
- for each invocation, run docker exec via SDK:
  - `nullclaw cron add <expression> <command>`
  - map `Invocation.Message` to shell command string (for example `nullclaw agent -m "<message>"`)
  - `Invocation.Name` and `Invocation.To` are not representable with current nullclaw cron CLI; warn and ignore
- capture stdout/stderr and include in errors
- add timeout context per exec
- idempotency requirement:
  - before add, query existing jobs with `nullclaw cron list`
  - skip add if same expression + command already exists
  - do not assume upstream duplicate-name protection (none in current CLI path)

#### HealthProbe

- at minimum, verify container running (consistent with existing lightweight drivers)
- if richer nullclaw-native probe is available, parse and return structured status

Core test coverage:

- registration and validate fail-closed behavior
- mount paths and generated files
- env + healthcheck config
- INVOKE shell-command builder behavior (`Invocation.Message` -> safe shell command)
- cron add arg builder behavior (positional expression/command)
- cron-list parser behavior for duplicate detection
- post-apply error handling (missing container ID / non-running container / exec failure)

Acceptance gate:

- `go test ./internal/driver/nullclaw -v` passes with no flaky tests.

## Upstream verification (resolved 2026-03-01)

1. **Health path/command**
   - Upstream documents `GET /health` as public gateway endpoint and uses gateway mode for long-running runtime.
   - There is no top-level `nullclaw health` command in the current CLI command table.

2. **Slack fields**
   - Slack config includes `bot_token` plus optional `app_token`, optional `signing_secret`, and `mode` (`socket` or `http`).
   - Runtime behavior requires:
     - `bot_token` always
     - `app_token` for `socket` mode (unless fallback path is active)
     - `signing_secret` for `http` mode

3. **Cron behavior**
   - CLI syntax is positional: `nullclaw cron add <expression> <command>`.
   - Current add path does not provide duplicate-name semantics and auto-generates IDs (`job-<n>`), so duplicate prevention must be handled by the driver if needed.

4. **CONFIGURE semantics**
   - Current top-level nullclaw CLI does not expose `config set`.
   - `CONFIGURE nullclaw config set ...` remains a Clawdapus DSL interpreted by the driver.

## Future extensions

- Add map-form `channel://...` surface routing support for nullclaw parity.
- Move from basic running-check `HealthProbe` to richer runtime diagnostics.
- If nullclaw later supports declarative cron config, shift INVOKE from exec-time registration to static config generation.
- Add optional compatibility mode to mirror `nullclaw onboard` output more closely.
